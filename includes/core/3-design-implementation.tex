
\subsection{Software specification}

\paragraph{ShAI} is Linux shell AI assistant, which leverages small local \glspl{llm} to
help user process information and enchance the smoothness of interaction with 
terminal.

\subsubsection{Application Architecture}

The application is designed as a monolithic system, separated into several components,
each responsible for a specific aspect of the overall functionality to ensure modularity
and per-module maintainability, testing and reuse. This modular approach reduces the
possibility of \gls{llm} hallucination by decomposing tasks into smaller, well-defined
components, and simplifies testing by allowing evaluation of each component independently.
The goal of each component is to provide a simple interface and functionality needed for
other components to work. The interaction between the application and the user is
primarily through a \gls{cli} interface.

\subsubsection{Requirements specification}

The target user of the application is a general Linux user who is familiar with
the command line. The application provides them with an assistant that can retrieve
information from local documents such as man manual pages database, as well as
other local textual resources in the form of text files running locally on the
user's machine without the need for internet connection.

The application is divided into several functional modules, each with distinct
user-facing capabilities. This will be detailed in the Architectural Design section below, with
diagrams showing the component structure and use cases.


\subsubsection{Architectural Design}

Based on the requirements above and the limitations of small \glspl{llm}
observed during development, the system is divided into focused, single-purpose
components. This modular approach prevents hallucination by constraining each
component's responsibility and enables thorough testing of individual modules.
Figure~\ref{fig:use-cases} presents the system's use cases and user
interactions.

\begin{figure}[ht]
  \centering
  \includegraphics[width=0.8\textwidth]{./img/shAI_Use_Cases.png}
  \caption{Use-case diagram for ShAI}\label{fig:use-cases}
\end{figure}

The application is structured around three main components, each designed to
handle specific aspects of the user interaction and information retrieval
process:

\begin{enumerate}
  \item \textbf{Natural Language vs. Shell Command Classifier (part of CLI
    Module)} --- will allow the system to distinguish between user commands
    and questions, making the interaction more seamless. It will use a
    combination of syntax checking and LLM-based classification to achieve
    high accuracy in distinguishing between shell commands and natural
    language queries.
  \item \textbf{Information Retriever (DB Module)}
    The task of this component is to maintain the document database and
    provide programmatic interface for dense search and managing
    documents. It will use a locally run embedding model to index
    these documents and enable contextual search.
  \item \textbf{Information Summarizer (RAG Module)}
    This component serves as master for \gls{rag} system, passing
    user query to \textbf{Information Retriever} and processing its outputs.
\end{enumerate}

More detailed system architecture is visualized in
Figure~\ref{fig:component-arch}, which shows the module structure of the

be modular and extensible, allowing for future enhancements and easy software

\subsection{Development Process}

The section describes the development-related aspects of the project, including
the implementation details of each component, testing strategies, and other
important information.

\subsubsection{Component 1. Natural Language vs. Shell Command Classifier}

\paragraph{Idea.}{}
The classifier component distinguishes between shell commands and natural language input using
LLM-based classification. If the input is a shell command, it can be executed directly
in the shell; if natural language, it engages the RAG pipeline for contextual answers.

\paragraph{Shell Integration Architecture.}{}
To capture user input without disrupting the shell environment, the implementation uses Bash
Readline hooks. The \texttt{get\_current\_user\_input} function is registered as a configurable
callback that extracts the current input buffer. Example hook for bash is shown
in Listing~\ref{lst:readline-hook}.

\begin{lstlisting}[,
  caption={Bash Readline Hook for Capturing User Input},
  label={lst:readline-hook},
  language=bash,
  style=code-listing,
  float
]
    extract_input() {
        echo -e "\n--- Captured Buffer ---"
        echo "Current input is: \$READLINE_LINE"
        echo "--- End Capture ---"
    }

    # Bind Ctrl-Y to execute that function
    bind -x '"\C-y": extract_input'
\end{lstlisting}

\paragraph{Generic CLI Support}{}
The design is generalized to support any command-line interface, not just Bash.
Main limiting factor is that each CLI requires registration of the hooks. In
this thesis, Bash is the primary implementation target.

\paragraph{Improvement to the Classifier}{}

To experiment with the classifier's performance, two versions are implemented
and tested:\label{sec:classifier-approaches}
\begin{enumerate}
  \item \textbf{LLM-Only Approach} will use the LLM alone to classify
    input using zero-shot prompting. This tests the generation
    model's language understanding capability on command vs.\ natural
    language distinction. Expected to work well for clear-cut cases
    but may struggle with ambiguous inputs.

  \item \textbf{Syntax Checker + LLM} will combine a deterministic
    syntax checker that validates input against Bash syntax rules. If
    the syntax is invalid it is passed to the LLM to be classified
    and processed. This hybrid approach is expected to achieve higher
    accuracy and better latency on well-formed shell commands while retaining
    LLM flexibility for natural language.
\end{enumerate}

\subsubsection{Component 2. Document retriever}

\paragraph{Idea.}{}
This component is responsible for managing document indexes and retrieving documents, relevant user queries.

\paragraph{Man pages processing for embedding database.}{}
For better context management and embedding generation, the pages will be
compiled to text format (see groff -Tascii and grotty). This will allow us to
extract the most relevant information without the noise of formatting and other
non-essential elements.

\paragraph{RAG Overhead and Proposed Solutions.}{}
One of the key challenges faced while implementing RAG in small \glspl{llm}
is the substantial prompt size it generates. This overhead negatively
impacts performance, precision, and latency. To address this, the
following two-step approach has been adopted:

\begin{enumerate}
  \item \textbf{Document Relevance Filtering:} An additional step is
    implemented where the assistant selects the most relevant
    documents based on easily identifiable beginnings (e.g., command
    names and short descriptions in manual pages). This is designed
    to optimize the initial retrieval process and ensure minimal
    context is sent to the \gls{llm}.
  \item \textbf{Iterative Prompt Processing:} To further optimize,
    the RAG system processes documents in small, manageable parts
    rather than in full. The results are aggregated after processing,
    reducing latency and improving precision.
  \item \textbf{Passage-level Retrieval:} Instead of passing entire documents to
    the \gls{llm}, the system retrieves and processes smaller passages from
    retrieved top-k documents.
\end{enumerate}

\paragraph{Testing. }{}
Two levels of evaluation are implemented for the document retriever component:

\begin{enumerate}
  \item \textbf{Test 1 (Document-level Precision@k):} Evaluates document retrieval accuracy
    using the testing collection of queries and expected documents with top@k
    metric. The test queries the indexed documents and measures how
    frequently the expected document appears in the top-k results.

  \item \textbf{Test 2 (Passage-level Precision@k):} Extends Test 1 by using passage-level
    extraction via \texttt{search\_in\_files\_dynamic}. This test splits documents into
    smaller passages and ranks them by relevance.

\end{enumerate}


\subsubsection{Component 3. Information Summarizer}

\paragraph{Idea.}{}
This component serves as the master for the RAG system, orchestrating the
retrieval and summarization process. It takes user queries, interacts with the
Information Retriever to fetch relevant documents, and processes the retrieved
information to generate summarized responses using techniques, described in the
Component 2 section.

\subsection{Deployment}

To deploy the \gls{llm} node a docker image is used hosting an Ollama image. This
ensures a controlled and isolated environment for running the local \glspl{llm}
effectively.

\paragraph{Service Checking Logic:} A dedicated service-checking routine
is implemented to verify that the \gls{llm} is running and available for
processing requests. This service runs at regular intervals and
includes retries and logging mechanisms to troubleshoot.

\subsection{Technology Stack}

This section provides an overview of the technologies and tools used in the
implementation, as well as the rationale behind their selection.

\subsubsection{Python, Python Libraries}

Python was chosen as the primary implementation language for this thesis due to
its simplicity, readability, and extensive ecosystem of libraries for machine
learning, and devise integration. smong such libraries, we use:

\begin{itemize}
  \item \texttt{openai}: Implementation of the OpenAI API client,
    used for interacting with \glspl{llm} services via HTTP requests.
    This standartized way to interact is ideal for architectural
    modularity and flexibility.
  \item \texttt{faiss-cpu}: A library for efficient similarity search
    and clustering of dense vectors, used for implementing the vector
    database for document retrieval. It provides fast indexing and
    querying capabilities, making it suitable for handling large
    collections of document embeddings.
  \item \textt{os, pathlib, subprocess, shutil}: Standard Python
    libraries for managing operating system resources, like
    filesystem operations, running subprocesses, and handling paths.
\end{itemize}

\subsubsection{Ollama}

Ollama\cite{ollama-web} is a local LLM service that enables running large
language models on consumer hardware without requiring cloud API calls. For this
thesis, Ollama was chosen as the primary LLM runtime over in-code alternatives
for several reasons:

\begin{itemize}
  \item \textbf{Simplicity and Ease of Use} --- Ollama provides a
    straightforward command-line interface for downloading,
    installing, and managing LLMs. This eliminates the need to
    implement model loading and inference logic directly in the application.

  \item \textbf{Model Flexibility} --- Ollama supports a wide variety
    of open-source models with different sizes and capabilities,
    allowing for easy experimentation with different architectures
    and parameter counts.

  \item \textbf{OpenAI API Compatibility} --- Ollama provides an
    OpenAI-compatible API endpoint, making it possible to test with
    both local (via Ollama) and remote (via OpenAI) models without
    changing application code. This flexibility was valuable for
    development and testing.

  \item \textbf{Resource Efficiency} --- Ollama is designed for local
    execution and allows both \gls{gpu}-enabled and \gls{cpu}-only
    inference, making it accessible for users without high-end hardware.
\end{itemize}

\subsubsection{\gls{faiss}}

For the RAG retrieval component, \gls{faiss}\cite{faiss2026} was
selected as the vector database for storing and retrieving document
embeddings. This choice was motivated by:

\begin{itemize}
  \item \textbf{Simplicity and efficiency} --- a minimal python API
    with no complex server setup or external dependencies, making it
    ideal for prototype development.

  \item \textbf{File-Oriented Storage} --- Annoy saves indices
    to disk files, allowing persistent storage of embeddings without
    requiring a running database server.
\end{itemize}
