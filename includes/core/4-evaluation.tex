\paragraph{Key Findings and Limitations}
Evaluation and testing of this component revealed critical limitations that
prevented deployment:

\begin{enumerate}
  \item \textbf{Poor Classification Accuracy:} Both the LLM-only and
    syntax-checker hybrid approaches exhibited accuracy below 70\% on
    ambiguous inputs, with particularly poor performance on edge
    cases common in real-world usage.
  \item \textbf{Unacceptable Latency:} Achieving higher accuracy
    required multiple LLM calls for validation and fallback attempts,
    resulting in total inference times of 1-3 seconds per input. This
    is unacceptable for real-time shell interaction.
  \item \textbf{Model Hallucination:} Small local \glspl{llm}
    (0.5B-4B parameters) demonstrated inconsistent behavior on
    classification tasks, sometimes misclassifying straightforward
    commands or generating plausible-sounding but incorrect classifications.
  \item \textbf{Shell-Specific Implementation Burden:} Achieving
    seamless integration across different shell environments (Bash,
    Zsh, Fish) requires shell-specific implementations that are
    difficult to maintain and test.
\end{enumerate}

\paragraph{Conclusion on Component 1}
This component remains a \textbf{proof-of-concept} and was not
pursued further. The fundamental limitation is that small local
\glspl{llm} cannot reliably classify ambiguous inputs in real-time
without either: (a) unacceptable latency through multiple calls, or
(b) extensive hardcoding of program paths that reduce
generalizability, (c) the shell wrapper itself becomes a bottleneck
and maintenance nightmare. The project focus was therefore shifted to
the \gls{rag} component, which proved more practical and effective.

\subsubsection{Component 2: Document Retriever}

\paragraph{Evaluation Approach}
The retriever is evaluated on its ability to return relevant
documents within top-k results,
using Precision@k at three levels of granularity:
\begin{enumerate}
  \item Document-level (whole files as retrieval unit)
  \item Document-to-passage-level (passage extraction from top-k documents)
  \item Passage-level (multiline passages as retrieval unit)
\end{enumerate}

\paragraph{Testing Collections}
\begin{itemize}
  \item \texttt{sample-10}: 10 documents, 150 queries (initial testing)
  \item \texttt{sample-100}: 100 documents, 300 queries
    (comprehensive evaluation)
\end{itemize}

\paragraph{Key Findings}
The document retriever component proved robust and practical. Vector
embedding-based retrieval using local models (e.g.,
Qwen3-Embedding-0.6B, Granite-Embedding-278M) achieved high
Precision@k with acceptable latency (typically 50-200ms per query).

document-level, providing more targeted information to the
summarizer. This component forms the reliable foundation of the
\gls{rag} system.

\paragraph{Key Findings}
The complete \gls{rag} system demonstrates practical performance for
\gls{cli} assistance:
\begin{itemize}
  \item Query latency: 250-700ms depending on model and document corpus size
  \item Memory footprint: 2-4 GB for 100+ documents with embedding indices
  \item Scalability: Tested with up to 200+ documents; performance
    remains acceptable
  \item User experience: The combination of fast retrieval + focused
    summarization provides relevant, accurate answers to command-line
    queries without the latency/accuracy issues of the classification component
\end{itemize}

\paragraph{Limitations}
The \gls{rag} system is constrained to document-based assistance and cannot
handle task-specific guidance or dynamic system introspection. It works well
for: explaining commands, providing usage examples, and retrieving
documentation. It is less suitable for: system diagnostics, real-time
monitoring, or complex decision-making tasks that require agent-like
capabilities.

\subsubsection{Why Agent-Based Approaches Failed}

During exploration, agentic approaches (where the \gls{llm} could iteratively
reason, use tools, and take actions) were tested to handle more complex tasks.
These experiments revealed fundamental limitations of small local \glspl{llm}:

\begin{enumerate}
  \item \textbf{Reasoning Chain Breakdown:} Multi-step reasoning
    required to plan and execute agent actions frequently broke down
    in small models, with incorrect intermediate conclusions
    cascading into failures.
  \item \textbf{Tool Use Unreliability:} Even with explicit tool
    definitions, small models struggled with proper tool invocation
    syntax and parameter selection, requiring heavy prompt
    engineering and validation logic.
  \item \textbf{Hallucination Under Uncertainty:} When agents faced
    unexpected states or error conditions, models tended to
    hallucinate plausible-sounding responses rather than correctly
    diagnosing issues or requesting clarification.
  \item \textbf{Unacceptable Latency:} Achieving higher accuracy
    required extensive prompting, validation loops, and fallback
    mechanisms, resulting in total latencies of 5-15 seconds per query.
\end{enumerate}

\paragraph{Conclusion on Agent-Based Approaches}
Agent-based and classification-focused approaches require significantly larger
models (10B+ parameters) or external \gls{llm} services to function reliably.
For local, privacy-preserving assistance on consumer hardware, the program paths
should rather be programmatically constrained and suited for specific tasks
(e.g., retrieval + summarization) rather than open-ended reasoning or tool use.
